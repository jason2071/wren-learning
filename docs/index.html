02 — ปัญหา

4 กับดักของฐานข้อมูลจริง

DB จริงมักรก: ความรู้กระจุกในหัวคนเก่า. คนใหม่/AI ที่ไม่รู้บริบท → เขียน SQL ผิดแบบ เงียบๆ (ได้เลข แต่ผิด). คลิกการ์ดเพื่อพลิกดูวิธี Wren แก้

03 — เทียบ

คำถามเดียวกัน เลขคนละเรื่อง

กดสลับ raw SQL ↔ Wren ดูว่าเลขเปลี่ยนยังไง (เลขจริงจาก demo seed: 200 ลูกค้า / 1200 orders)

04 — องค์ประกอบ

4 ชิ้นที่ทำงานร่วมกัน

บริบทที่เคยอยู่ในหัวคน ถูกย้ายมาเป็นไฟล์ config ที่ทั้งคนและ AI อ่านได้

📖models/

พจนานุกรมชื่อ

name = ชื่อที่ถาม · table_reference = ตารางจริง · columns ที่ประกาศเท่านั้นที่เห็น (ไม่ประกาศ = masked)

🔗relationships.yml

แผนที่ join

บอกว่าตารางต่อกันยังไง (orders → customers MANY_TO_ONE) → ถามข้ามตารางได้ AI ไม่ join มั่ว

📜instructions.md

กติกาธุรกิจภาษาคน

นิยามที่ SQL บอกไม่ได้: active คืออะไร, revenue นับยังไง, status เลขไหนแปลว่าอะไร

🧠.wren/memory/

ความจำ

เก็บ context + คู่ "คำถาม↔SQL" เก่า เป็น vector — ค้นด้วยความหมาย ไม่ใช่ตรงตัวอักษร

05 — Memory

Memory — สมุดโน้ตอัจฉริยะ

หัวใจที่ทำให้ Wren ยิ่งใช้ยิ่งฉลาด — มี 3 คำสั่งหลัก

fetch

เปิดสมุดดูบริบทที่เกี่ยวข้อง

ดึง context (schema + นิยาม) ที่เกี่ยวกับคำถาม ให้ AI ใช้ก่อนเขียน SQL

recall

เคยมีคนถามแบบนี้ไหม

หาคู่ "คำถาม↔SQL" เก่าที่ความหมายใกล้คำถาม (few-shot ให้ AI)

store

จดคำตอบที่ถูกไว้

บันทึกคู่ "คำถาม↔SQL" ใหม่ ให้ครั้งหน้า recall เจอ

subcommands ทั้งหมด (ตรวจกับ CLI แล้ว): index · fetch · store · recall · list · forget · load · reset · status. wren memory index default โหลด queries.yml + instructions.md เข้า memory อัตโนมัติ
06 — อ่านผล

distance: ยิ่งต่ำ ยิ่งใกล้

recall เรียงผลตามระยะห่างความหมาย. คล้าย ≠ ถูก — ต้องดู distance แล้วตรวจ SQL เอง

$ wren memory recall -q "สมัครวันนี้กี่คน"
0.735
ลูกค้ามีกี่คน
SELECT COUNT(*) FROM customers
1.367
List all customers
SELECT * FROM customers LIMIT 100
1.398
Total customer_id in orders
SELECT SUM(customer_id) FROM orders
⚠️ ใกล้สุด 0.735 = "ลูกค้ามีกี่คน" แต่นั่นคือนับทั้งหมด ไม่ใช่วันนี้ — ระยะ 0.7+ ยังถือว่าห่าง (ตรงจริงจะ ~0.1–0.3). ไม่มีของตรง → เขียนเอง แล้ว store ไว้
07 — วงจร

recall → write → store

ครั้งแรกอาจไม่มีอะไรให้ recall → สอนมันด้วย store. ใช้ไปเรื่อยๆ คลังโต → recall แม่นขึ้น → AI ผิดน้อยลง

STEP 1
recall

หาของเก่าที่คล้าย — เจอ dist สูง = ยังไม่มีของตรง

STEP 2
write + run

เขียน SQL เอง รันจริง ตรวจเลขให้ถูก

STEP 3
store

จดคู่ "คำถาม↔SQL" ที่ถูก — ครั้งหน้า recall เจอ dist ต่ำ

09 — ลงมือ

เจอคำถามใหม่ ทำยังไง

หลักตัดสิน 1 ประโยค: คำถามต้องใช้ field / join / นิยาม ที่ layer ยังไม่มีไหม? — มีครบ ข้าม เขียน SQL ได้เลย · ขาด เพิ่ม ที่ชั้นที่ขาดก่อน. คลิกการ์ดพลิกดูว่าต้องทำอะไร

ทางลัด — ไม่ต้องรันทุกคำสั่ง

เลือกตามความมั่นใจ. recall / dry-plan = เครื่องมือ debug ใช้ตอนไม่ชัวร์เท่านั้น ปกติข้าม

ระดับ 1 · เร็วสุด
1 คำสั่ง

แค่อยากได้คำตอบ · field มั่นใจครบ
wren --sql "..."

ระดับ 2 · เก็บไว้
+ store

อยากใช้ซ้ำ
wren memory store --nl --sql

ระดับ 3 · ครบ
git track

append queries.yml
→ build && index

ลำดับบังคับ: context build ต้องก่อน memory index เสมอ (index อ่าน mdl.json) · เพิ่ม column/relationship → build ก่อนรัน SQL ที่ใช้ field ใหม่ · store เข้า memory ตรง ไม่ต้อง build
rule of thumb: ตอบเฉยๆ = 1 คำสั่ง · เก็บถาวร = +store หรือ +queries.yml/build/index. รายละเอียดเต็ม + worked example → docs/wren_new_question_recipe.md
